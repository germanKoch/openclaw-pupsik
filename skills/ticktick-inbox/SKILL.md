# Разбор инбокса TickTick

Разобрать все задачи в Inbox по GTD / «Техника джедая»: переформулировать, распределить по проектам, расставить приоритеты — с учётом целей пользователя.

## Контекстные файлы

Перед началом загрузи контекст с диска (если файлы существуют):

- **Проекты:** `~/.openclaw/workspace/ticktick-inbox/projects.md`
- **Цели:** `~/.openclaw/workspace/ticktick-inbox/goals.md`

Эти файлы сохраняются между сессиями и обновляются по итогам каждого запуска.

---

## Алгоритм

### Фаза 0. Загрузить контекст

1. Прочитать `projects.md` и `goals.md` если существуют.
2. Получить актуальный список проектов: `ticktick.get_projects`
3. Если в `projects.md` есть проекты, которых уже нет в TickTick — удалить их из файла.
4. Если есть новые проекты (которых нет в файле) — перейти в Фазу 1.

---

### Фаза 1. Разобраться в проектах

Для каждого проекта из `ticktick.get_projects`:

- Если смысл проекта уже известен из `projects.md` — пропустить.
- Если проект непонятен (название абстрактное, одно слово, аббревиатура и т.п.) — **спросить пользователя**:
  > «У тебя есть проект **«{name}»** — расскажи в 1-2 словах, о чём он? Что туда складывается?»
- После получения ответа — записать смысл в файл.

Формат `projects.md`:
```
# TickTick Projects

## {project_name}
**Смысл:** {одна фраза — о чём проект}
**Тип:** [Работа / Личное / Финансы / Здоровье / Обучение / Другое]
**Примечание:** {опционально}

---
```

Сохранить файл: `~/.openclaw/workspace/ticktick-inbox/projects.md`

---

### Фаза 2. Выявить цели пользователя

Если `goals.md` не существует или старше 7 дней — выполнить полное сканирование.

1. Получить задачи из всех проектов (кроме Inbox): `ticktick.get_project_tasks` для каждого project_id.
2. Искать признаки целей:
   - Задачи с формулировкой «Достичь / Стать / Запустить / Заработать / Научиться / Завершить»
   - Задачи без конкретного действия (скорее намерение, чем шаг)
   - Задачи-вехи с далёкими дедлайнами
   - Задачи в проектах типа «Цели», «Goals», «Планы», «2025», «Год»
3. Составить список целей и сохранить в файл.

Формат `goals.md`:
```
# Цели пользователя
_Обновлено: {дата}_

## Высокоуровневые цели
- {формулировка} — проект: {project}
- ...

## Горизонты (если нашёл)
- **Этот месяц:** ...
- **Этот год:** ...
- **Долгосрочно:** ...
```

Сохранить файл: `~/.openclaw/workspace/ticktick-inbox/goals.md`

---

### Фаза 3. Разобрать инбокс

Получить задачи: `ticktick.get_project_tasks project_id=inbox121828750`

Для каждой задачи определить:

**Название по GTD:**
- Начинать с глагола (конкретное действие)
- Следующий конкретный шаг (next action), не абстракция
- Измеримый результат — прочитал название, знаешь что делать
- Если задача на >2 часов — декомпозировать

Примеры переформулирования:
- «разобраться с налогами» → «Собрать расходы за 2025 по категориям для налогового вычета»
- «позвонить врачу» → «Записаться на первичную консультацию по брекетам»
- «купить подарок» → «Выбрать и заказать подарок на день рождения [имя]»
- «бенчмарки» → «Собрать список 15 бенчмарков и сравнить метрики в таблице»

**Описание (content):**
- Добавить `Next action:` с конкретным первым шагом
- Добавить полезные инсайты/уточнения, если очевидны из контекста
- Сохранять существующее описание — дополнять, не затирать

**Проект:** подобрать из существующих, используя `projects.md` для понимания смысла каждого. Если не уверен — НЕ переносить, вынести в вопросы.

**Приоритет** (0 = none, 1 = low, 3 = medium, 5 = high) — с учётом целей из `goals.md`:
- **High:** задача прямо продвигает одну из текущих целей, ИЛИ горящий дедлайн, ИЛИ блокирует другие
- **Medium:** важно, но не срочно; косвенно связано с целями
- **Low:** nice-to-have, когда-нибудь; не связано ни с одной целью
- **None:** справочные заметки, идеи

При выставлении приоритета явно указывай в отчёте, если задача связана с целью:
> ↑ High — продвигает цель «Запустить X до марта»

**Дедлайн:** если из контекста очевидна дата. Ставить all-day (без конкретного времени).

**Теги-нотация (предложения):**

Проверить название и описание задачи на признаки метаинформации:

- **Дедлайн** — фразы: «до [дата]», «дедлайн», «к [дата]», «сдать», «крайний срок», «не позднее» → предложить тег `DUE/D_YYYY.MM.DD`
- **Человек** — имя собственное в контексте задачи («с Ксенией», «для Ивана», «обсудить с [имя]») → предложить тег `PERSON/P_ИМЯ`
- **Earliest start** — «не раньше [дата]», «актуально с [дата]», «начать с [дата]» → предложить тег `NOT_EARLIER/N_YYYY.MM.DD`

Предложения **добавлять в блок вопросов** (Фаза 4), не применять автоматически.

Если задача имеет существующий `DUE/D_*` тег → считать дедлайн уже установленным, не предлагать повторно.

**Приоритет с учётом дедлайна (FR-013):**
- Если задача имеет тег `DUE/D_*` (существующий или предложенный) с дедлайном ≤ 7 дней от сегодня → назначить **High** приоритет
- Указать в отчёте: `↑ High — горящий дедлайн [дата]`

---

### Фаза 4. Два прохода: уверенные + вопросы

**Уверенные кейсы** — сразу обновлять через `ticktick.update_task`:
- title — переформулированное название
- content — дополненное описание с next action
- priority — приоритет (с учётом целей)
- dueDate — если очевидна

**Неоднозначные кейсы** — НЕ трогать, собрать в список вопросов:
- task_id, текущее название
- В чём сомнение
- 1-2 варианта проекта
- Предложения тегов из Фазы 3 (если есть): добавить в вопрос строку «Предлагаю добавить тег: `DUE/D_2026.04.15` — добавить? (да/нет)»

---

### Фаза 5. Советы по организации

После обработки инбокса — проанализировать общую структуру задач и проектов. Давать советы только если видишь реальные проблемы, не ради советов.

Проверить:
- **Мёртвые задачи:** есть ли просроченные задачи >30 дней без активности? → Предложить архивировать или пересмотреть
- **Перегруженные проекты:** проект >50 задач без структуры? → Предложить разбить на подпроекты или добавить теги
- **Осиротевшие задачи:** задачи без проекта (кроме Inbox), без приоритета и без дедлайна → Предложить ревью
- **Дублирующиеся проекты:** похожие по смыслу проекты → Предложить объединить
- **Цели без задач:** если в `goals.md` есть цель, но в TickTick нет ни одной задачи, явно к ней привязанной → Сообщить
- **Inbox как хранилище:** если в инбоксе были задачи старше 14 дней → Предложить настроить привычку ежедневного разбора

Формат совета:
```
⚠️ {Проблема одной фразой}
   Что нашёл: ...
   Рекомендация: ...
```

---

### Фаза 6. Ограничения API

- `update_task` **не переносит** задачи между проектами (project_id не меняется через update). Перенос: создать новую задачу в целевом проекте + удалить старую. Или оставить рекомендацию для ручного переноса.
- НИЧЕГО не удалять автоматически.
- Если задача потенциально неактуальна — не удалять, вынести в вопросы.

---

### Фаза 7. Отчёт

```
📋 Обработано автоматически: N задач
- «старое название» → «новое название» (приоритет ↑ High — горящий дедлайн 10 апр / цель: X)
- ...

❓ Вопросы (нужно уточнение): M задач

1) [task_id]
   Текущее: «название»
   Сомнение: ...
   Варианты: ПРОЕКТ_A / ПРОЕКТ_B
   Предлагаю добавить тег: `DUE/D_2026.04.15` — добавить? (да/нет)

📦 Рекомендации по переносу (вручную): K задач
- «задача» → проект X

🧊 Заморожено (не раньше [дата]): K задач
- «задача» — заморожена до: 1 июн 2026 (проект: X)
- «задача» — заморожена до: 20 апр 2026 (проект: Y) ⚠️ горящий дедлайн 10 апр

🎯 Цели — обновлено в goals.md
- Найдено целей: ...

💡 Советы по организации:
⚠️ ...
```

Секция 🧊 включается только если есть задачи с `NOT_EARLIER/N_*` тегом с датой в будущем.
Задача в 🧊 секции НЕ обрабатывается в обычном потоке разбора (не переформулируется, не переносится) — только отображается.
