# Финансовый анализ

Сформировать детальный финансовый отчёт за месяц: доходы, расходы по категориям, индекс финансового здоровья, глубокий анализ изменений, сравнение с предыдущими месяцами, тренды, аномалии, конкретные практики экономии и рекомендации. Данные берутся из ZenMoney.

---

## Хранение данных

Данные хранятся в **двух местах** с разными целями:

### Диск: `~/.openclaw/workspace/financial_analysis/`

Полное хранилище: кэш транзакций, промежуточные расчёты, детальная история по месяцам, маппинг категорий. Используется для быстрого доступа без повторных API-вызовов и для глубокого анализа.

Структура:

```
~/.openclaw/workspace/financial_analysis/
├── config.json                     # маппинг категорий, настройки
├── cache/
│   ├── accounts.json               # кэш счетов (обновлять раз в запуск)
│   ├── categories.json             # кэш категорий (обновлять раз в запуск)
│   └── transactions/
│       ├── 2025-01.json            # транзакции за январь (сырые, из API)
│       ├── 2025-02.json            # транзакции за февраль
│       └── ...
├── reports/
│   ├── 2025-01.md                  # готовый отчёт за январь (Markdown)
│   ├── 2025-02.md                  # готовый отчёт за февраль
│   └── ...
└── history/
    ├── 2025-01.json                # полная аналитика за январь
    ├── 2025-02.json                # полная аналитика за февраль
    └── ...
```

### TickTick: заметка `financial_analysis` в проекте CLAWD

Только **компактная сводка** для быстрого просмотра Юзером. Без сырых данных, без промежуточных расчётов. Содержит: ключевые метрики по месяцам, индекс здоровья, тренд, последний отчёт.

- **Проект:** `CLAWD` (project_id: `698c924c8f08f21724f5daba`)
- **Заметка:** `financial_analysis` — найти task_id по названию при первом запуске
- Если заметка не существует — создать.

## Режимы работы

### Полный ежемесячный отчёт (по умолчанию)

Запуск 1-го числа (крон) или по ручному запросу. Выполняются все фазы.

### Еженедельный мини-отчёт

Запуск по запросу «как дела с финансами», «weekly», `mode=weekly`, или между 2-м и 28-м числом месяца по неформальному запросу о финансах.

Формат (компактный):

```
📊 Неделя {N} ({дата_от} – {дата_до})

Потрачено за неделю: {X} руб.
Темп расходов: {Y} руб./мес (при сохранении темпа)
Бюджет месяца (прогноз): {средний доход} руб.
Запас: {доход − прогноз расходов} руб. ({+/−}%)

🔥 Топ-3 категории за неделю:
1. {cat} — {X} руб.
2. {cat} — {X} руб.
3. {cat} — {X} руб.

{Если темп расходов > среднего дохода:}
⚠️ При текущем темпе ты потратишь на {X} руб. больше дохода. Притормози с {топ-категория}.
```

Для мини-отчёта: выполнить только Фазу 0 (контекст) → Фазу 1 (данные за текущую неделю) → мини-агрегация → вывод. Остальные фазы пропустить.

## Алгоритм полного отчёта

### Фаза 0. Загрузить контекст

1. **Создать структуру директорий** (если не существует):
   ```
   mkdir -p ~/.openclaw/workspace/financial_analysis/{cache/transactions,reports,history}
   ```

2. **Загрузить конфиг** (`config.json`):
    - Если файл существует — прочитать маппинг категорий, task_id TickTick, known_payees.
    - Если не существует — создать пустой конфиг, task_id будет найден ниже.

3. **Найти заметку TickTick** (если task_id не в конфиге):
    - Вызвать `ticktick.get_project_with_data projectId=698c924c8f08f21724f5daba`
    - Найти задачу с названием `financial_analysis`
    - Если не найдена — создать: `ticktick.create_task` с `title=financial_analysis`, `projectId=698c924c8f08f21724f5daba`
    - Сохранить task_id в `config.json`

4. **Определить период отчёта:**
    - Если запуск 1-го числа (крон) → **предыдущий календарный месяц** целиком.
    - Если запуск в любой другой день (ручной вызов) → **текущий месяц с 1-го числа по вчера** включительно. Пометить `is_partial=true`.
    - Запомнить `date_from`, `date_to`, `period` (YYYY-MM).

5. **Загрузить историю с диска:**
    - Прочитать все файлы из `history/*.json` — это полная аналитика предыдущих месяцев.
    - Отсортировать по `period` для хронологического порядка.
    - Запомнить количество месяцев с историей (для определения доступных фаз).

---

### Фаза 1. Получить данные из ZenMoney

**Принцип: сначала проверить кэш, потом API.**

1. **Счета:**
    - Вызвать `zenmoney.get_accounts` (всегда свежие — балансы меняются).
    - Сохранить в `cache/accounts.json`.
    - Составить маппинг account_id → валюта.

2. **Категории:**
    - Если `cache/categories.json` существует и младше 7 дней → использовать кэш.
    - Иначе: вызвать `zenmoney.get_categories`, сохранить в `cache/categories.json`.

3. **Транзакции текущего периода:**
    - Если `cache/transactions/{period}.json` существует И `is_partial=false` → использовать кэш.
    - Если `is_partial=true` → всегда загружать заново (данные неполного месяца обновляются).
    - Иначе: вызвать `zenmoney.get_transactions` с `date_from`, `date_to`, `limit=500`.
    - Вызвать `zenmoney.get_budgets` с теми же датами.
    - Сохранить в `cache/transactions/{period}.json` (транзакции + бюджеты + метаданные).

4. **Транзакции прошлых месяцев** (для Фазы 2.7 — глубокий анализ):
    - НЕ загружать сейчас. Будут загружены по требованию в Фазе 2.7 из кэша или API.

5. **Классификация транзакций:**
    - Для каждой транзакции определить валюту по account_id.
    - Разделить на три группы:
        - **Расходы**: сумма отрицательная, не перевод между своими счетами
        - **Доходы**: сумма положительная, не перевод между своими счетами
        - **Переводы**: между собственными счетами (исключить из доходов/расходов)

---

### Фаза 2. Агрегация и анализ

**Для каждой валюты отдельно:**

1. **Итоги:**
    - Общий доход = сумма всех доходных транзакций
    - Общие расходы = сумма всех расходных транзакций (абсолютное значение)
    - Переводы = сумма всех переводов (абсолютное значение)
    - Баланс = доход − расходы

2. **Расходы по категориям:**
    - Сгруппировать расходы по категории
    - Для каждой категории: сумма, доля от общих расходов (%), количество транзакций, средний чек
    - Транзакции без категории → группа «Без категории»
    - Отсортировать по сумме убывания

3. **Топ-5 категорий расходов:** первые 5 из отсортированного списка.

4. **Сравнение с предыдущим месяцем (MoM):**
    - Загрузить `history/{prev_period}.json` с диска.
    - Если есть:
        - Для дохода, расходов, баланса: абсолютное изменение и процент
        - Для каждой категории расходов: процент изменения
        - Направление: ↑ если выросло >5%, ↓ если упало >5%, → если ±5%
    - Если нет: пропустить сравнение, написать «нет данных за предыдущий месяц».

5. **Аномалии:**
    - Для каждой категории вычислить среднюю сумму одной транзакции.
    - Найти транзакции, превышающие 2× среднюю для своей категории.
    - Для каждой аномалии: дата, сумма, получатель (payee), категория.

### Фаза 2.5. Индекс финансового здоровья

**Выполнять всегда.**

#### Маппинг категорий

Проверить `config.json` → `category_mapping`. Если отсутствует или появились новые категории из ZenMoney:
- Классифицировать все категории по типам:
    - **basic**: жильё/аренда, продукты, транспорт (общественный), здоровье, коммунальные, связь, образование (базовое)
    - **neutral**: одежда, курсы/развитие, бытовые подписки, домашние животные
    - **discretionary**: рестораны/кафе, развлечения, хобби, алкоголь, такси (сверх разумного), стриминг, электроника (не необходимая)
- Нераспознанные → neutral.
- Сохранить/обновить в `config.json`.

#### Формула (0–100 баллов)

1. **Норма сбережений** (0–35):
    - savings_rate = (доход − расходы) / доход × 100%
    - >30% → 35 | 20–30% → 28 | 10–20% → 20 | 0–10% → 10 | <0% → 0

2. **Доля неразумных трат** (0–30):
    - discretionary_ratio = дискреционные / общие расходы × 100%
    - <15% → 30 | 15–25% → 22 | 25–35% → 15 | 35–50% → 8 | >50% → 0

3. **Стабильность расходов** (0–20):
    - При 2+ месяцах истории: CV общих расходов за последние N месяцев.
    - CV <10% → 20 | 10–20% → 15 | 20–35% → 10 | >35% → 5
    - Без истории → 10 (нейтраль)

4. **Отсутствие аномалий** (0–15):
    - 0 → 15 | 1–2 → 10 | 3–5 → 5 | >5 → 0

#### Интерпретация

- 80–100: 🟢 «Отличный месяц. Финансовая дисциплина на высоте.»
- 60–79: 🟡 «Нормальный месяц. Есть точки для улучшения.»
- 40–59: 🟠 «Средний месяц. Расходы вышли из-под контроля.»
- 0–39: 🔴 «Тяжёлый месяц. Нужна корректировка.»

#### Историческое сравнение (при 2+ месяцах)

- Шкала: «Твой индекс: 72/100 🟡 (прошлый: 81, средний: 75)»
- Ранг: «{N}-й по качеству из {M} проанализированных.»
- Тренд: «68 → 75 → 81 → 72 (↓ ухудшение после хорошей серии)»

### Фаза 2.7. Глубокий анализ изменившихся категорий

**Выполнять при 2+ месяцах истории.**

#### Алгоритм

1. **Выбор категорий:** отобрать те, где расходы изменились >30% относительно среднего. Максимум 5 (3 выросших + 2 упавших).

2. **Для каждой выбранной категории:**

   a) Транзакции текущего месяца — из `cache/transactions/{period}.json`, отфильтровать по категории.

   b) Транзакции за предыдущий месяц:
    - Проверить `cache/transactions/{prev_period}.json`.
    - Если нет — вызвать `zenmoney.get_transactions date_from={prev_start} date_to={prev_end} limit=500`, сохранить в кэш.
    - Отфильтровать по категории.

   c) Транзакции за «лучший» месяц (при 3+ месяцах истории):
    - Определить месяц с минимальными расходами в этой категории из `history/*.json`.
    - Проверить `cache/transactions/{best_period}.json`.
    - Если нет — вызвать API, сохранить в кэш.
    - Отфильтровать по категории.

3. **Диагностика причины изменения:**

    - **Рост частоты:** количество транзакций +20%, средний чек ±10%
    - **Рост среднего чека:** количество ±20%, средний чек +15%
    - **Крупная разовая трата:** одна транзакция >40% расходов категории
    - **Новый получатель:** новый payee >25% расходов категории
    - **Комбинация:** и частота, и чек выросли
    - **Снижение (позитив):** расходы −20%+

4. **Рецепт из лучшего месяца:**
   → «В {месяц} ты тратил {X} руб. Что было по-другому: {Y} транзакций (сейчас {Z}), средний чек {A} руб. (сейчас {B}), получатели: {список}. Экономия при возврате: ~{разница} руб./мес.»

### Фаза 2.8. Анализ получателей

**Выполнять всегда.**

1. **Топ-10 получателей** по сумме: payee, сумма, количество, категория.

2. **Новые получатели** (при 2+ месяцах): payees, которых не было в `history/{prev}.json` → `top_payees`.

3. **Исчезнувшие получатели** (при 3+ месяцах): payees из 3+ предыдущих месяцев подряд, пропавшие сейчас.

4. **Рекуррентные платежи:** payees с суммой ±5% каждый месяц.

5. **Обновить known_payees** в `config.json` (payee → категория) для будущих `suggest_category`.

### Фаза 2.9. Детекция импульсных покупок

**Выполнять всегда.** Если ничего не найдено — секцию не выводить.

Эвристики:
1. **По времени** (если доступно): 23:00–05:00
2. **По кластеризации:** 3+ покупок в одной дискреционной категории за 1 день
3. **По получателю:** маркетплейсы (Wildberries, Ozon, AliExpress, Lamoda) в дискреционных категориях
4. **По отклонению:** >3× среднего чека в дискреционной категории

---

### Фаза 3. Тренды и кросс-корреляции

**Выполнять при 3+ файлах в `history/`.**

1. **Тренды по категориям** (топ-10):
    - Собрать суммы из `history/*.json`
    - ↑ Рост: +5% два месяца подряд | ↓ Снижение: −5% два подряд | → Стабильно

2. **Кросс-корреляции:**
    - Пары из топ-10, движущиеся в одном/противоположном направлении 3+ раз.
    - Максимум 3 наблюдения.

3. **Сезонность** (при 6+ месяцах):
    - Сравнить с тем же месяцем прошлого года (если есть `history/{same_month_prev_year}.json`).

### Фаза 4. Оценка и рекомендации

#### 4.1. Оценка накоплений

**Обязательная секция.**

- savings_rate = (доход − расходы) / доход × 100%
- Динамика: X% → Y% → Z% (из `history/`)
- Оценка:
    - >20%: «Отличный месяц. Ты откладываешь {X}% — выше рекомендуемых 20%.»
    - 10–20%: «Неплохо — {X}% ушло в накопления ({сумма} руб.).»
    - 0–10%: «Впритык — только {X}% ({сумма} руб.). Съели бюджет: {топ-3}.»
    - <0%: «Минус: расходы больше доходов на {сумма} руб. Причины: {топ-3}.»
- Средняя норма сбережений за всё время (при 2+ месяцах).

#### 4.2. Общая оценка месяца

**Обязательная секция.** Тон — дружелюбный, но прямой.

Обязательно включить:
- Индекс здоровья + сравнение со средним
- Ранг месяца (при 2+ мес.)
- Главный позитив
- Главный вызов (с суммой)
- Цель на следующий месяц (одна, конкретная, измеримая)

#### 4.3. Детальные рекомендации

5–8 конкретных рекомендаций. Каждая ОБЯЗАНА ссылаться на категории, суммы, даты.

Типы:
1. Перерасход (>20% к прошлому)
2. Аномалия (крупная транзакция)
3. Позитивный тренд (снижение)
4. Бюджет (превышение)
5. Растущий тренд (3+ мес.)
6. Структура (>30% всех расходов)
7. Подписки (всегда упоминать если есть)
8. Падение сбережений (2+ мес.)
9. Импульсные покупки
10. Новый крупный получатель

НЕ давать общие советы. Только конкретика с цифрами.

#### 4.4. Конкретные практики экономии

**Обязательная секция.** Топ-3 проблемных категории → конкретный совет.

##### База практик

**Продукты:**
- Заказ через Яндекс.Лавку/Самокат с подпиской (−10–15% + меньше импульсных)
- Список покупок (−20–30% спонтанных)
- Сравнение цен через Едадил (разница 15–25%)
- Средний чек >3000 → проверить долю готовой еды (2–3× дороже)

**Рестораны/кафе:**
- Правило 50%: заменить каждый второй визит готовкой
- Бизнес-ланчи вместо меню (разница 1.5–2×)
- Подписка на доставку при {N}+ заказах/мес

**Транспорт:**
- >10 такси → подписка Яндекс.Go Plus (−10–20%)
- Такси vs каршеринг на поездках >15 мин
- Проездной вместо разовых (−20–40%)

**Подписки:**
- Аудит: App Store / Google Play → Подписки → отменить неиспользуемое >30 дней
- Семейные планы (−40–60%)
- Объединение в экосистему (Яндекс.Плюс ~300 руб.)

**Одежда:**
- Маркетплейсы vs офлайн
- Правило 72 часов (покупки >5000 руб.)
- Сезонные распродажи

**Развлечения:**
- Стоимость часа (кино 150 руб./ч vs концерт 1667 руб./ч)
- Бесплатные альтернативы

**Здоровье:**
- Проверить покрытие ДМС
- Дженерики (−30–70%), проверка на apteka.ru

**Связь:**
- Проверка тарифа раз в 6 мес.
- >700 руб./мес → сравнить на tarifer.ru

Для каждой практики: персонализировать числами и рассчитать экономию.

#### 4.5. Прогноз на следующий месяц

**При 3+ месяцах истории.**

- Скользящее среднее за 3 мес. + тренд → прогноз по категориям.
- Суммарные прогнозные расходы vs ожидаемый доход.
- Категории под угрозой роста (тренд >+10%/мес).

|------------|-------|-----------|

### Фаза 6. Сохранить результаты

#### 6.1. Сохранить на диск

1. **Полная аналитика** → `history/{period}.json`
    - Формат: см. «Диск: history/YYYY-MM.json» выше.
    - Если файл за период уже существует — перезаписать.

2. **Готовый отчёт** → `reports/{period}.md`
    - Точная копия Markdown, отправленного Юзеру.

3. **Обновить конфиг** → `config.json`
    - `last_run`, обновлённый `category_mapping`, `known_payees`.

#### 6.2. Отправить отчёт в Telegram

**Обязательный шаг.** Отправить полный Markdown-отчёт из Фазы 5 Хозяину в Telegram.

- Отчёт может быть длинным — разбить на части по 4000 символов если нужно, отправить последовательно.
- Не сокращать и не резюмировать — отправить весь отчёт целиком.
- После отправки отчёта — отправить отдельным коротким сообщением: «✅ История сохранена в TickTick (financial_analysis)»

#### 6.3. Обновить TickTick

1. Прочитать текущий `content` заметки.
2. Сформировать компактную сводку (формат — см. «TickTick: содержимое заметки» выше).
3. **Полностью перезаписать** content заметки свежей сводкой.
    - `ticktick.update_task` с `taskId`, `id`, `projectId=698c924c8f08f21724f5daba`, `content=<сводка>`

Сводка включает:
- Текущий статус (индекс, тренд, лучший месяц, средний)
- Таблица последних 6 месяцев (доход, расход, сбережения%, индекс)
- Текущие цели (максимум 5)
- Регулярные платежи (список)
- Топ-3 категории с пометками роста
- Потенциал экономии

**Правила для TickTick:**
- Максимум 6 месяцев в таблице — старые удалять.
- Цели: не более 5. Новая цель из 4.2 добавляется. Выполненные помечаются [x], удаляются через месяц.
- Регулярные платежи: обновлять из `recurring_payees`.
- **Никакого JSON. Никаких сырых данных. Только человекочитаемый Markdown.**

## Кэширование: правила

| Данные | Хранение | Инвалидация |
|--------|----------|-------------|
| Счета | `cache/accounts.json` | Каждый запуск (балансы меняются) |
| Категории | `cache/categories.json` | Если старше 7 дней |
| Транзакции (полный месяц) | `cache/transactions/YYYY-MM.json` | Никогда (месяц закрыт) |
| Транзакции (неполный месяц) | `cache/transactions/YYYY-MM.json` | Каждый запуск |
| Полная аналитика | `history/YYYY-MM.json` | Перезаписывается при повторном анализе |
| Конфиг | `config.json` | Обновляется каждый запуск |
| Готовые отчёты | `reports/YYYY-MM.md` | Перезаписывается при повторном анализе |

**Выгода кэширования:**
- Фаза 2.7 (глубокий анализ) загружает транзакции прошлых месяцев С ДИСКА, а не через API.
- Повторный запуск за тот же закрытый месяц не делает API-вызовов для транзакций.
- Все тренды/корреляции считаются из `history/`, без API.

## Ограничения

- НЕ создавать, не обновлять, не удалять транзакции в ZenMoney. Только чтение.
- НЕ давать общие финансовые советы без привязки к данным.
- НЕ конвертировать валюты — каждую валюту отдельно.
- НЕ показывать пустые секции отчёта.
- НЕ изобретать практики экономии — только из базы или эвристики на данных.
- НЕ хранить сырые данные и JSON в TickTick — только компактную сводку.
- Если ZenMoney недоступен — сообщить и прекратить.
- Если `get_transactions` вернул 500 (лимит) — предупредить о неполноте данных.
