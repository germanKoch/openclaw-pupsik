# Feature Specification: TickTick Tag Notation System

**Feature Branch**: `004-ticktick-tag-notation`
**Created**: 2026-03-04
**Status**: Draft
**Input**: Система тегов-нотации для schedule-агента — дедлайны, earliest-start, связь с людьми

---

## Overview

Вводится система структурированных тегов в TickTick, которая позволяет агенту хранить метаинформацию о задачах: реальный дедлайн (отдельно от даты выполнения), самую раннюю возможную дату начала, и связь с конкретным человеком. Агент обязан создавать, читать и использовать эти теги во всех операциях с задачами.

---

## Tag Notation Reference

TickTick поддерживает вложенные теги через `/`. Три типа тегов-нотации:

| Тип | Родительский тег | Формат дочернего | Пример | Семантика |
|-----|-----------------|-----------------|--------|-----------|
| Дедлайн | `DUE` | `D_YYYY.MM.DD` | `DUE/D_2026.12.31` | Реальный дедлайн (дата без времени) |
| Earliest start | `NOT_EARLIER` | `N_YYYY.MM.DD` | `NOT_EARLIER/N_2026.06.01` | Задача неактуальна до этой даты |
| Человек | `PERSON` | `P_ИМЯ` | `PERSON/P_KSENIIA` | Задача связана с человеком |

**Ключевое разделение для DUE:**
- `dueDate` задачи = дата **выполнения** (scheduling date, когда запланировано)
- `DUE/D_*` тег = **реальный дедлайн** (когда задача должна быть готова)

Это позволяет запланировать задачу на вторник, зная что дедлайн — пятница.

## User Scenarios & Testing

### User Story 1 — Создание задачи с дедлайном (Priority: P1)

Пользователь просит агента создать задачу с явным дедлайном. Агент сохраняет дедлайн в теге `DUE/D_*`, дату выполнения — в `dueDate`.

**Why this priority**: Базовая операция — без корректного создания тегов система не работает.

**Independent Test**: Создать задачу через агента с явным дедлайном → в TickTick проверить тег `DUE/D_YYYY.MM.DD`.

**Acceptance Scenarios**:

1. **Given** пользователь указал дату выполнения И дедлайн, **When** агент создаёт задачу, **Then** `dueDate` = дата выполнения, тег `DUE/D_YYYY.MM.DD` присутствует
2. **Given** пользователь указал только дедлайн без отдельной даты выполнения, **When** агент создаёт задачу, **Then** `dueDate` = дедлайну, тег `DUE/D_YYYY.MM.DD` добавлен
3. **Given** пользователь указал только дату выполнения без дедлайна, **When** агент создаёт задачу, **Then** тег `DUE/*` не добавляется
4. **Given** пользователь обновляет существующую задачу, добавляя дедлайн, **When** агент обновляет, **Then** тег `DUE/D_*` добавляется к существующим тегам

---

### User Story 2 — Недельное планирование с учётом дедлайнов (Priority: P1)

При формировании плана на неделю агент сканирует теги `DUE/*` и выделяет задачи с дедлайном ≤7 дней в секцию «Горящие дедлайны» — независимо от их `dueDate`.

**Why this priority**: Главная ценность — дедлайны не теряются среди задач с разными датами выполнения.

**Independent Test**: Создать задачу с `DUE/D_<через 3 дня>` и `dueDate` на следующей неделе → запросить план → задача в секции горящих дедлайнов.

**Acceptance Scenarios**:

1. **Given** задача с `DUE/D_2026.03.07` (через 3 дня) и `dueDate` на следующей неделе, **When** агент строит план на неделю, **Then** задача выделена в секции «Горящие дедлайны»
2. **Given** задача с `DUE/D_2026.04.30` (через месяц), **When** агент строит план на неделю, **Then** задача не попадает в секцию горящих дедлайнов
3. **Given** несколько задач с `DUE/*` дедлайнами в течение недели, **When** агент строит план, **Then** они отсортированы по дедлайну, ближайший первым
4. **Given** задача с `DUE/D_<сегодня>`, **When** агент строит план, **Then** задача помечена как «дедлайн сегодня»

---

### User Story 3 — Подавление NOT_EARLIER задач при планировании (Priority: P2)

При планировании агент исключает задачи с `NOT_EARLIER/N_*` если дата ещё не наступила. При разборе инбокса — информирует о заморозке, не предлагает к планированию.

**Why this priority**: Снижает шум — пользователь не видит задачи, которые нельзя начать.

**Independent Test**: Создать задачу с `NOT_EARLIER/N_<через 14 дней>` → запросить план → задача отсутствует.

**Acceptance Scenarios**:

1. **Given** задача с `NOT_EARLIER/N_2026.03.20` (через 16 дней), **When** агент формирует план на неделю, **Then** задача не включается в план
2. **Given** задача с `NOT_EARLIER/N_2026.03.01` (дата прошла), **When** агент формирует план, **Then** задача обрабатывается как обычная
3. **Given** разбор инбокса, задача имеет `NOT_EARLIER` в будущем, **When** агент формирует отчёт, **Then** задача выведена отдельно с пометкой «заморожена до [дата]»
4. **Given** задача с `NOT_EARLIER` в будущем И `DUE` в течение 7 дней, **When** агент строит план, **Then** задача попадает в «Горящие дедлайны» с дополнительной пометкой о заморозке

---

### User Story 4 — Создание и фильтрация по PERSON (Priority: P2)

Пользователь создаёт задачу с упоминанием человека. Агент добавляет тег `PERSON/P_ИМЯ`. По запросу фильтрует задачи по человеку.

**Acceptance Scenarios**:

1. **Given** пользователь указывает имя при создании задачи, **When** агент создаёт задачу, **Then** добавляется тег `PERSON/P_ИМЯ` (латиница, верхний регистр)
2. **Given** задача связана с несколькими людьми, **When** агент создаёт задачу, **Then** добавляется по тегу `PERSON/P_*` для каждого
3. **Given** задачи с тегом `PERSON/P_KSENIIA`, **When** пользователь просит «покажи всё связанное с Ксенией», **Then** агент фильтрует по тегу и показывает список

---

### User Story 5 — Интеграция с ticktick-inbox при разборе входящих (Priority: P2)

При разборе инбокса (скил `/ticktick-inbox`) агент замечает признаки дедлайна или имён и предлагает добавить теги — в рамках GTD-практик.

**Acceptance Scenarios**:

1. **Given** задача в инбоксе с текстом «сдать до 15 апреля», **When** агент разбирает, **Then** предлагает добавить `DUE/D_2026.04.15` (в блоке вопросов)
2. **Given** задача «Обсудить контракт с Ксенией», **When** агент разбирает, **Then** предлагает добавить `PERSON/P_KSENIIA`
3. **Given** агент предлагает тег нотации, **When** формирует запрос, **Then** НЕ добавляет автоматически — только с подтверждения

---

### Edge Cases

- Несколько `DUE/*` тегов на одной задаче → брать ближайший, предупреждать об аномалии
- `NOT_EARLIER` позже `DUE` (начало позже дедлайна) → предупреждение при создании/обнаружении
- Невалидная дата в теге (`D_2026.13.01`) → игнорировать при парсинге, сообщить пользователю
- Имя с пробелом → `P_IVAN_PETROV` (пробел = `_`)
- Задача одновременно в «горящих дедлайнах» и скрыта `NOT_EARLIER` → показывать с двойной пометкой

## Requirements

### Functional Requirements

- **FR-001**: Агент добавляет тег `DUE/D_YYYY.MM.DD` при создании/обновлении задачи когда пользователь указывает дедлайн
- **FR-002**: Агент добавляет тег `NOT_EARLIER/N_YYYY.MM.DD` когда пользователь указывает дату earliest start
- **FR-003**: Агент добавляет тег `PERSON/P_ИМЯ` когда пользователь упоминает конкретного человека
- **FR-004**: Все теги нотации создаются как subtag родительского (`DUE`, `NOT_EARLIER`, `PERSON`)
- **FR-005**: При недельном планировании агент сканирует `DUE/*` всех задач и выделяет задачи с дедлайном ≤7 дней в секцию «Горящие дедлайны»
- **FR-006**: Секция «Горящие дедлайны» выводится первой в плане, задачи отсортированы по дате дедлайна
- **FR-007**: Задачи с `NOT_EARLIER/N_*` в будущем не включаются в планы на день/неделю
- **FR-008**: При разборе инбокса задачи с `NOT_EARLIER` в будущем выводятся отдельно с пометкой о заморозке
- **FR-009**: При разборе инбокса агент предлагает (но не добавляет автоматически) теги нотации при наличии признаков дедлайна или имён
- **FR-010**: Агент расшифровывает теги нотации в человекочитаемый вид при показе задач
- **FR-011**: Невалидные теги нотации игнорируются с уведомлением пользователя
- **FR-012**: При конфликте `NOT_EARLIER > DUE` агент предупреждает пользователя
- **FR-013**: Задача с `DUE/*` в пределах 7 дней получает высокий приоритет при разборе инбокса (согласно FR-005 скила ticktick-inbox)

### Key Entities

**Tag (тег нотации)**

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | `DUE` / `NOT_EARLIER` / `PERSON` | Тип тега |
| `fullTag` | string | `DUE/D_2026.12.31` |
| `date` | `YYYY.MM.DD` | Для DUE и NOT_EARLIER |
| `name` | string | Для PERSON (латиница верхний регистр) |

**Task (расширенная модель)**

| Поле | Источник | Описание |
|------|----------|----------|
| `dueDate` | TickTick field | Дата выполнения (scheduling) |
| `deadline` | тег `DUE/D_*` | Реальный дедлайн |
| `earliestStart` | тег `NOT_EARLIER/N_*` | Earliest start date |
| `persons` | теги `PERSON/P_*` | Связанные люди |

## Success Criteria

- **SC-001**: 100% задач с явно указанным пользователем дедлайном содержат тег `DUE/D_*` после обработки агентом
- **SC-002**: При недельном планировании задачи с дедлайном ≤7 дней всегда выделены в отдельную секцию
- **SC-003**: Задачи с `NOT_EARLIER` в будущем не появляются в планах на текущий период (0 ложных включений)
- **SC-004**: 100% создаваемых тегов нотации соответствуют формату (`D_YYYY.MM.DD`, `N_YYYY.MM.DD`, `P_ИМЯ`)
- **SC-005**: Агент никогда не показывает сырые теги вида `DUE/D_2026.04.30` — всегда в расшифрованном виде

---

## Scope

**Включено:**
- Три типа тегов: DUE, NOT_EARLIER, PERSON
- Обновление скила `daily-planner` (секция горящих дедлайнов, фильтрация NOT_EARLIER)
- Обновление скила `ticktick-inbox` (предложение тегов при разборе, учёт дедлайнов при приоритизации)
- Обновление `agents/schedule/AGENTS.md` и `agents/schedule/TOOLS.md` с описанием нотации

**Исключено:**
- Автоматическое добавление тегов без явного сигнала от пользователя
- Автоматический перенос `dueDate` при приближении дедлайна
- Синхронизация с Google Calendar или другими системами
- Визуальный UI вне чата

---

## Dependencies & Assumptions

**Зависимости:**
- TickTick API поддерживает вложенные теги в формате `parentTag/childTag`
- schedule-агент существует с инструментами TickTick
- Скилы `daily-planner/SKILL.md` и `ticktick-inbox/SKILL.md` существуют и обновляются в рамках этой фичи

**Допущения:**
- Теги в TickTick API приходят как массив строк в поле `tags`
- Имена в `PERSON/*` — латиница, верхний регистр; пробел заменяется `_`
- Формат даты — `YYYY.MM.DD` с точками
- «Горящий дедлайн» = дедлайн в течение 7 дней от текущей даты
- Родительские теги создаются TickTick автоматически при первом использовании subtag
